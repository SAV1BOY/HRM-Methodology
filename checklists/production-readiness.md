# Production Readiness Checklist

> Use this checklist before deploying any LLM-powered feature to production. Covers robustness, security, performance, monitoring, fallbacks, testing, and documentation. Every item marked with **(critical)** must pass.

---

## Robustness

- [ ] **(critical)** Prompt produces correct output on 20+ representative test inputs
- [ ] **(critical)** Prompt handles edge cases gracefully (empty input, very long input, special characters)
- [ ] Output format is consistent across diverse inputs (no format drift)
- [ ] Prompt is resilient to minor input variations (typos, different phrasings)
- [ ] Temperature and sampling parameters are tuned and documented
- [ ] Model version is pinned (not "latest") to prevent regression on updates
- [ ] Retry logic handles transient API failures with exponential backoff
- [ ] Timeout is set appropriately for the expected response time

## Security

- [ ] **(critical)** Prompt injection defenses are in place (sandwich defense, instruction hierarchy, or input sanitization)
- [ ] **(critical)** No secrets, API keys, or credentials are embedded in prompts
- [ ] **(critical)** User input is sanitized before injection into prompts
- [ ] System prompt cannot be extracted via adversarial user input
- [ ] PII in prompts is minimized or anonymized
- [ ] Data sent to the LLM API complies with data residency requirements
- [ ] Output filtering prevents generation of harmful, illegal, or policy-violating content
- [ ] Rate limiting is applied per user/session to prevent abuse
- [ ] Audit logging captures prompt-response pairs for security review

## Performance

- [ ] **(critical)** End-to-end latency meets the UX requirement (< {target}ms for P95)
- [ ] Token usage is monitored and within budget
- [ ] Caching is implemented for repeated or similar queries
- [ ] Prompt length is optimized (no unnecessary context inflation)
- [ ] Batch processing is used where real-time response is not required
- [ ] Streaming is enabled for user-facing responses that exceed 2 seconds
- [ ] Concurrent request limits are configured to prevent API throttling
- [ ] Cost per request is calculated and within the per-unit budget

## Monitoring

- [ ] **(critical)** Logging captures: prompt hash, input summary, output summary, latency, token count, model version
- [ ] **(critical)** Alerting is configured for: error rate spike, latency spike, cost anomaly
- [ ] Quality metrics are tracked (user ratings, downstream task success rate)
- [ ] Output format compliance is monitored (JSON parse success rate, schema validation)
- [ ] Hallucination rate is measured via sampling and human review
- [ ] Model drift detection compares output distributions over time
- [ ] Dashboard exists for real-time visibility into system health

## Fallbacks

- [ ] **(critical)** A fallback response exists for when the LLM API is unavailable
- [ ] **(critical)** A fallback exists for when the model returns malformed output
- [ ] Graceful degradation is implemented (e.g., simpler prompt, smaller model, cached response)
- [ ] Human escalation path exists for low-confidence or high-stakes outputs
- [ ] Circuit breaker pattern prevents cascading failures when API is degraded
- [ ] Default/safe responses are defined for each failure mode

## Testing

- [ ] **(critical)** Unit tests cover the prompt template with mocked LLM responses
- [ ] **(critical)** Integration tests verify end-to-end flow with real API calls (on a schedule)
- [ ] Regression test suite captures previously-fixed failure cases
- [ ] Evaluation dataset exists with ground-truth labels for automated scoring
- [ ] A/B testing framework is ready for prompt version comparisons
- [ ] Load testing confirms the system handles expected peak traffic
- [ ] Adversarial testing has been performed (prompt injection, jailbreak attempts)
- [ ] Cross-model testing validates the prompt works if the model is swapped

## Documentation

- [ ] **(critical)** Prompt purpose and expected behavior are documented
- [ ] **(critical)** All prompt parameters and their valid ranges are documented
- [ ] Model selection rationale is documented (why this model for this task)
- [ ] Known limitations and failure modes are documented
- [ ] Runbook exists for common operational issues
- [ ] Change log tracks prompt version history with dates and reasons
- [ ] Architecture diagram shows the LLM integration in the broader system
- [ ] Cost projections are documented for expected and peak usage

## Compliance and Ethics

- [ ] Output accuracy has been validated by domain experts
- [ ] Bias testing has been performed across relevant demographic groups
- [ ] User disclosure is in place ("this response was generated by AI")
- [ ] Data retention policies comply with applicable regulations (GDPR, CCPA, etc.)
- [ ] Content moderation is applied to both inputs and outputs
- [ ] Human oversight is in place for high-stakes decisions
